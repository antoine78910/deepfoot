# Visifoot 2.0 — Backend flow & summary generation

## 1. What happens when you click "Analyze the match with AI"

```
Frontend (POST /predict { home_team, away_team })
    → Backend load_match_context()
    → Backend predict_all() (Poisson)
    → Optional: OpenAI quick_summary + scenario_1
    → Response (probs, xG, form, bars, summary, scenario)
```

---

## 2. Data loading (`load_match_context`)

**Priority:** API-Football (if `API_FOOTBALL_KEY` is set) → Supabase (if configured) → fallback demo data.

### 2.1 If API-Football is used

1. **Resolve team names to IDs**  
   `resolve_team_name_to_id(home_team)` and same for away. Uses an in-memory cache of teams from all configured leagues (filled via `/teams?league=X&season=Y`).

2. **Fetch last 10 fixtures per team**  
   `GET /fixtures?team={id}&season={year}&status=FT`  
   Only finished matches (FT). Season = current European season (e.g. 2025 for 2025-26).

3. **From these fixtures we compute:**
   - **Goals for / against** (last 5): for each fixture, whether the team was home or away, we take goals scored and conceded.
   - **Form** (last 5): W/D/L per match.
   - **H2H**: we filter the home team’s fixtures to those vs. the away team, then count home wins, draws, away wins.  
   If there are **no H2H matches**, the code still runs; in `build_comparison_pcts` we now use **50% / 50%** for H2H so it doesn’t skew the rest.

4. **Averages and lambdas**
   - `compute_goals_avg(goals_for, goals_against)` → average goals scored and conceded.
   - `compute_lambda_home_away(...)`: Dixon–Robinson style strength:
     - Attack strength ∝ goals for, defense ∝ 1/goals against.
     - `lambda_home = (home_attack * away_defense) / league_avg`
     - `lambda_away = (away_attack * home_defense) / league_avg`  
   League average is 2.7 (configurable). Lambdas are clipped to [0.2, 4.0].

5. **Comparison percentages (for the bars)**  
   `build_comparison_pcts(...)` returns:
   - **Attack** %: share of “goals for” (home vs away).
   - **Defense** %: share of “1 / goals against” (better defense → higher %).
   - **Form** %: share of “form points” (W=3, D=1, L=0) over total.
   - **H2H** %: home share of H2H “points” (win=1, draw=0.5).  
   If any of these has **no data** (e.g. no H2H), that dimension is set to **50% / 50%** so it doesn’t distort the overall bar.

6. **Overall %**  
   Average of the 5 dimensions above (attack, defense, form, h2h, goals).  
   All of this is **deterministic**: no AI, only maths on fixtures and goals.

### 2.2 If Supabase is used (no or failed API-Football)

- Teams are identified by **slug** (normalized name or API team id stored as text).
- **results** table: last N rows where `home_team_id` or `away_team_id` = slug → goals for/against, form.
- **h2h** table: row `(home_team_id, away_team_id)` → home_wins, draws, away_wins.  
Same formulas then: goals avg → lambda → comparison pcts (with 50/50 when data is missing).

### 2.3 Fallback

- If neither API-Football nor Supabase gives data, default demo goals/form and H2H (e.g. 1-1-1) are used so the pipeline still returns a valid response.

## 3. Predictions (`predict_all` — Poisson model)

**File:** `backend/app/ml/poisson.py`

- **Input:** `lambda_home`, `lambda_away` (expected goals per team for this match).
- **Model:** Independent Poisson:  
  `P(home = i, away = j) = Poisson(i | λ_home) * Poisson(j | λ_away)`  
  Grid is built for scores 0..8 (configurable `max_score_goals`).

From the grid we derive:

| Output | How it’s computed |
|--------|-------------------|
| **xG home / away / total** | λ_home, λ_away, λ_home + λ_away (rounded). |
| **1X2** | Sum grid: Home = sum(i>j), Draw = sum(i=j), Away = sum(i<j). |
| **BTTS Yes/No** | Sum of P(i,j) for i≥1, j≥1 → Yes; remainder → No. |
| **Over/Under** | For lines 0.5, 1.5, 2.5, 3.5: Over = sum of P(i+j > line). |
| **Exact scores** | All (i,j) sorted by P(i,j), top 5 returned. |

**No AI here:** only Poisson maths from the two lambdas.

## 4. Summary and scenario (AI or not?)

**File:** `backend/app/services/openai_summary.py`

- **quick_summary** and **scenario_1** are generated by **OpenAI** (model `gpt-4o-mini`) when:
  - `OPENAI_API_KEY` is set in your env (e.g. `.env.local`), and
  - The call does not throw (e.g. rate limit, network).

- **Prompt:**  
  We build a short text (teams, xG, 1X2 probs, form labels) via `build_prompt_context()`.  
  The model is asked for a short neutral summary (2–3 sentences) and one paragraph for a likely scenario, in the same language as the team names.

- **If there is no key or the call fails:**  
  You get fallback strings, for example:
  - `"Summary based on stats and form (AI summary unavailable)."`
  - `"Scenario based on expected goals and form (AI scenario unavailable)."`  
  So if you **don’t see any requests in your OpenAI dashboard**, it means either:
  - `OPENAI_API_KEY` is not set or not loaded, or
  - An exception is thrown (e.g. key invalid, network error) and the fallback is used.

So: **the “quality” text can be either real AI (OpenAI) or the fixed fallback.**  
To use real AI: set `OPENAI_API_KEY` in `.env.local` and ensure the backend can reach the API; then you should see usage in the OpenAI dashboard.

## 5. Full response shape

The API returns:

- Teams, optional league/date/venue.
- **xG** home, away, total.
- **1X2** probabilities (home %, draw %, away %).
- **BTTS** Yes/No %.
- **Over/Under** per line (0.5, 1.5, 2.5, 3.5).
- **Top 5 exact scores** with probabilities.
- **Form**: last 5 W/D/L, W-D-L string, form label (e.g. “Great form”).
- **Comparison bars**: attack, defense, form, h2h, goals, overall (all %).
- **quick_summary** and **scenario_1**: from OpenAI if key set and call succeeds, otherwise fallback text.

## 6. Summary table

| Step | Source | AI? |
|------|--------|-----|
| Team IDs | API-Football cache or Supabase | No |
| Fixtures / results | API-Football or Supabase | No |
| Goals avg, lambda, comparison pcts | Formulae in `data_loader` + `features` | No |
| 1X2, xG, BTTS, Over/Under, exact scores | Poisson in `poisson.py` | No |
| quick_summary, scenario_1 | OpenAI `gpt-4o-mini` or fallback text | Yes (only if key set and call succeeds) |

So: **everything except the short summary and scenario is deterministic (stats + Poisson).**  
The only part that uses the OpenAI API (and can show up in your OpenAI logs) is the summary and scenario when the key is set and the request succeeds.

## 7. Why don’t I see any OpenAI API logs?

If the text you see is high quality but **you don’t see requests in your OpenAI dashboard**, it usually means:

- **`OPENAI_API_KEY` is not set** (or not loaded from `.env` / `.env.local`).  
  Then the backend uses the **fallback** strings, for example:  
  `"Summary based on stats and form (AI summary unavailable)."`  
  and  
  `"Scenario based on expected goals and form (AI scenario unavailable)."`  
  So the “summary” and “scenario” are fixed text, not generated by the model.

- **The request failed** (invalid key, network, rate limit).  
  In that case the code catches the exception and returns the same fallback strings.

To have **real AI-generated** summary and scenario and see usage in OpenAI:

1. Set `OPENAI_API_KEY` in `backend/.env.local`.
2. Restart the backend.
3. Run an analysis again; you should then see calls to `gpt-4o-mini` in your OpenAI usage/logs.
